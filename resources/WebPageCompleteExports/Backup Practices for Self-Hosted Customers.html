Backup Practices for Self-Hosted Customers


1986 Views Last updated : Mar 7, 2024 Internal, Self-hosted Copy Permalink

Overview

ServiceNow customers can backup their environments using several different methods and available softwares. This article will cover some of the useful tips and techniques deployed in the hosted environment which may be helpful in planning backup strategy. These are only references, and not recommendations or requirements for your environment, although we highly recommend frequent backups in compliance with your company's internal process/policy.

Audience

Typical audiences of this document include system administrators, database administrators (DBAs), network engineers, architects, and helpdesk management who are planning to perform the backups on the database servers.

Risk and Cautions

  • Backups should be regularly tested to ensure a working backup copy is available whenever needed.
  • Backups should be initially attempted on a sub-production environment to ensure they do no impact the running environment.
  • If any 3rd Party backup software is used, check the documentation to ensure it supports the required database version. This check should be repeated during any database upgrades.
  • If any issues are encountered during the backups, database or backup vendors should be consulted for technical support.

Backup practices

Backup practices may change from organisation to organisation. Below are some of the practices followed in the ServiceNow hosted environment:

  • Full backup are performed at least once a week.
  • Incremental backups (based on the full backup) are performed every day. The incremental backups consume around 10% of the space usually consumed by a full backup, though each depends on the usage of the instance.
  • Chain incremental backups (from LSN based on another incremental backup) are not performed, as they have proved less reliable, slower to restore and don’t save significant storage.
  • ServiceNow hosted environment may occasionally request a full backup earlier than expected. Full backups are usually performed before and after Instance upgrade, and before any major database operation where we expect the differential backup to grow larger.
  • The ServiceNow hosted environment backup service detects a backup as required according to backup policy. Older backups are automatically deleted according to the cleanup policy - 7 days of daily backups (incremental), 30 days of weekly backups (full). Retention policy will differ with your company's internal process/policy.
  • Consider backup lower priority compared to any other impactful operation such as patching, upgrade, server maintenance. Backups may fail while in progress and they are retried if they run into any issue, locks, conflict or performance impact.
  • Always test backups and restores with the introduction of any new version of database etc to avoid any unknown bugs/issues.

LSN = log sequence number*
Below are the commands which are used for the backups:

MySQL Backups:

ServiceNow hosted environment uses innobackupex for MySQL backups. More details about innobackupex can be found here -> Link. The Prerequisites for using innobackupex can be found here - Link. Generally there are following main steps to take a backup. For each backup command, there are many different options/switches that can be used and they vary from situation to situation. These can be also be used in conjunction with other tools based on backup strategy, SLA or specific requirements:
1. Full backup
2. Incremental Backup
3. Prepare/Apply Base backup.
4. Prepare/Apply L1..Ln (if more increments)

Sample commands are like:

Full/Base/Level 0 backup:
innobackupex  --user=<USER> --host=<HOST> --port=<port> --password=<PASSWORD> --no-timestamp --no-version-check  /glide/backup/mak/inc
 
Incremental backup/Level 1
innobackupex --user=<USER> --host=<HOST> --port=<port> --password=<PASSWORD> --no-timestamp --no-version-check  /glide/backup/mak/incL1 --incremental -incremental-basedir=/glide/backup/mak/inc

Prepare L0/Base Backup:
innobackupex --user=<USER> --host=<HOST> --port=<port> --password=<PASSWORD> --apply-log --redo-only /glide/backup/mak/inc
 
Prepare L1/Ln Backup:
innobackupex --user=<USER> --host=<HOST> --port=<port> --password=<PASSWORD> --apply-log  /glide/backup/mak/inc --incremental-dir=/glide/backup/mak/incL1

For ServiceNow hosted environment, backup commands are issued like (for reference only):

Base/L0/Full:
innobackupexsnc --defaults-file=/glide/mysqld/mysqlbackuptest_3401_small/etc/backup.cnf --no-version-check \
--user=dbi_admin --throttle=50000 --parallel=1 --stream=xbstream --tmpdir=/glide/backup/temp/mysqlbackuptest_3401_db168020_3_2021-04-20_0642057 \
--no-lock /glide/backup/temp/mysqlbackuptest_3401_db168020_3_2021-04-20_0642057 2>/glide/backup/log/mysqlbackuptest_3401_db168020_3_2021-04-20_0642057.log 
| qpress -ioT4L1 '' | openssl enc -e -aes-256-cbc -md sha1 -pass file:********

Incremental/L1:
 innobackupex --defaults-file=/glide/<Database_path>/etc/backup.cnf --no-version-check \
 --incremental --incremental-lsn=******** --user=<user> --throttle=50000 --parallel=1 \
 --stream=xbstream --tmpdir=/glide/backup/<some_path>_********** --lock-wait-timeout=300 \
 --lock-wait-threshold=86400 --lock-wait-query-type=update \
 --slave-info /glide/backup/<some_path>_********** 2>/glide/backup/log/<some_path>_**********.log \
 | qpress -ioT4L1 '' | openssl enc -e -aes-256-cbc -md sha1 -pass file:********

The command will result in a binary stream that can be directed to a mounted filesystem (NFS or otherwise), piped through SSH to another server, or streamed to blob storage (e.g. Azure Blob or S3). Qpress with those parameters offers a decent tradeoff between disk usage and decompression speed. Details on qpress can be found HERE. ServiceNow use streaming to backup server so --stream=xbstream option is used but this is all optional. Finally, we encrypt each backup with a unique key to make the backups more secure. 
 
Full backups omit the --incremental and --incremental-lsn parameters.

MariaDB Backups:

ServiceNow hosted environment uses mariabackup for MariaDB backups. Mariabackup, a fork of Persona XtraBackup is included with MariaDB 10.1.23 and later. More details can be found HERE.

Similar to MySQL (innobackupex) commands described above:

Full backup:

mariabackup -hlocalhost -P<port> -usbtest -psbtest --backup --target-dir=/glide/backup/<some_path>_********** \
   --host=127.0.0.1 --port=<port> --binlog-info=AUTO --no-lock --throttle=50000 \
   2> /glide/backup/<some_path>_**********log/backup.log
 

Incremental backup:

mariabackup -hlocalhost -P<port> -usbtest -psbtest --backup --target-dir=/glide/backup/<some_path>_********** \
   --host=127.0.0.1 --port=<port> --binlog-info=AUTO --no-lock --throttle=50000 \
   --incremental-basedir=/glide/backup/<some_path>_********** 2> \
   /glide/backup/<some_path>_**********log/backup.log


Prepare(apply-log) BASE/L0 Backup.

/glide/mariadb/10.2.27/bin/mariabackup -hlocalhost -P3400 -usbtest -psbtest  --prepare --target-dir=/backup/nfs/backuptest_3400_large

Prepare(apply-log) L1 Backup:

/glide/mariadb/10.2.27/bin/mariabackup -hlocalhost -P3400 -usbtest -psbtest --prepare --target=/backup/nfs/backuptest_3400_large --incremental-dir=/backup/nfs/backuptest_3400_large/L1


For ServiceNow hosted environment, backup commands are issued like (for reference only):

Full/Base Backup
mariabackup --defaults-file=/glide/<Database_path>/etc/backup.cnf --no-version-check \
 --user=<user> --throttle=50000 --parallel=1 --stream=xbstream \
    --tmpdir=/glide/backup/<some_path>_********** --no-lock --backup \
    --target-dir=/glide/backup/<some_path>_********** \
    2> /glide/backup/log/<some_path>_**********.log | qpress -ioT4L1 '' \
    --format\=xbs.qp --lock\=false \
Incremental Backup
mariabackup --defaults-file=/glide/<Database_path>/etc/backup.cnf --no-version-check \
 --user=<user> --throttle=50000 --incremental-lsn=123456  --parallel=1 --stream=xbstream \
    --tmpdir=/glide/backup/<some_path>_********** --no-lock --backup \
    --target-dir=/glide/backup/<some_path>_********** \
    2> /glide/backup/log/<some_path>_**********.log | qpress -ioT4L1 '' \
    --format\=xbs.qp --lock\=false \

Note: ServiceNow backup use various options alongwith compression and encryption which are additional tools and not part of the backups themselves. To use any of these options, please be aware of their full use and implications to avoid any unwanted results.

More details on the mariabackup options can be found HERE.


Oracle Backups:

In the ServiceNow hosted environment, Oracle RMAN tool is used to backup the Oracle databases. Please refer the Oracle RMAN documentation for a full explanation of the RMAN utility. RMAN full backup runs weekly, and archive log backup runs every 4 hours. Archive logs older than 2 days which are backed up once are deleted from the disk. 

Full backup:
Here is code snippet for full backup (for reference only):

backupstatus="OK"
DATE=`date +%Y_%m_%d`
BACKUP_TAG=FULL`date +%Y%m%d`T`date +%H%M%S`
 
rman target / <<EOF
  backup database tag=$BACKUP_TAG;
  exit
EOF
 
 
rmanstatus=`sqlplus -s "/ as sysdba" <<\EOF
            set heading off feedback off verify off
            select status from
              (select status from v$rman_status where operation = 'BACKUP' and object_type = 'DB FULL' order by start_time desc)
            where rownum = 1;
            exit
EOF`
if [ $rmanstatus = 'COMPLETED' ]; then
   backupstatus="OK"
else
   backupstatus="Fail"
fi


Archive log backup:
Here is code snippet for archive log backup(for reference only):

#Set DAYS=2 default if the parameter not supplied
if [ -z "$DAYS" ]; then
   DAYS=2
fi
 
rman target / <<EOF
  run {
  backup archivelog all not backed up 1 times;
  delete noprompt archivelog all backed up 1 times to disk completed before 'sysdate-$DAYS';
  }
 exit
EOF
 
Obsolete backups:
Obsolete backups are checked and deleted 4 hourly,
rman target / <<EOF
  delete noprompt obsolete;
  exit
EOF

 
ServiceNow also perform weekly full export dump of the application schema for Oracle databases.
Example:

expdp "db_backup/<PASSWORD>" directory=$DUMPDIR dumpfile=$FNAME1\_%U.dmp logfile=$FNAME1.log schemas=$app_schema FILESIZE=200G
